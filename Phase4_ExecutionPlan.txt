Phase 4 执行规划（工作区临时文本）

一、关于当前“bug”的初步判断

1. 你的猜想有一定价值，值得优先验证。
2. 当前 shader 的主观察射线来自 Game 摄像机位置与方向，而不是 Scene 视图摄像机。
3. 如果你是在 Scene 面板里切换到一个并非 Game 摄像机的观察角度截图，那么你看到的画面可能是：
   - 代理立方体/对象在 Scene 视图里被从另一个角度观察
   - 但 shader 内部仍按 Game 摄像机位置计算 ray origin / ray dir
4. 这种情况下，确实可能出现“可视化观察角度”和“实际 raymarch 相机角度”不一致，进而表现出遮挡关系、切面朝向、命中区域看起来不对。
5. 但这通常更像“观察与渲染基准不一致”问题，不一定是最终运行时 bug。
6. 另外，之前看到的“切面颜色泄漏到外壳”的问题也依然可能同时存在，两者并不互斥。

二、验证优先级

建议按这个顺序验证：

1. 先验证当前异常是否主要由 Scene/Game 视角不一致导致。
2. 再验证 cut-face 判定是否确实有语义泄漏。
3. 只有在这两件事分离清楚后，再继续软阴影和体积光。

三、下一轮建议执行内容

第 1 步：加入一个用于验证的轨道摄像机脚本

目标：

- 不依赖 Scene 面板随意转视角
- 用 Game 摄像机直接围绕目标物体观察
- 保证“你看到的画面”和“shader 用来发射 ray 的摄像机”是同一个

建议控制方式：

- 以一个目标点为中心轨道旋转
- A / D：水平绕 Y 轴旋转
- W / S：竖直绕目标俯仰旋转
- 可选：
  - Q / E：缩放半径
  - R：重置角度
  - Shift：加速旋转

脚本行为建议：

- 挂在 Main Camera 上
- 暴露：
  - target
  - orbitCenterOffset
  - radius
  - yaw
  - pitch
  - yawSpeed
  - pitchSpeed
  - minPitch / maxPitch
  - minRadius / maxRadius
- LateUpdate 中更新相机位置
- 始终 LookAt 目标中心

第 2 步：用 Game 视图复现同一现象

目标：

- 判断这个现象是不是只在 Scene 观察时出现

验证方法：

1. 停用手动 Scene 观察作为判断依据。
2. 只在 Game 视图中使用轨道摄像机观察同一组切开的 piece。
3. 记录以下情况：
   - 如果问题在 Game 视图里明显减轻或消失，说明先前判断受到了 Scene/Game 视角不一致的影响。
   - 如果问题在 Game 视图里仍稳定存在，说明 shader 内部切面分类/法线/阴影逻辑确实有问题。

第 3 步：如有必要，再加入调试显示模式

目标：

- 把“观察角度问题”和“shader 语义错误”彻底分开

建议后续可加的调试显示：

- 显示 base shape
- 显示 cut-face mask
- 显示 plane-dominance mask
- 显示 normal override 区域
- 显示 SDF soft shadow 值
- 显示仅主光阴影

四、对阶段 4 的修正后执行顺序

原计划：

1. SDF 软阴影
2. 动态体积光

修正后建议：

1. 先验证 Scene/Game 视角一致性
2. 再修正 cut-face 识别与切面视觉语义
3. 再上 SDF 软阴影
4. 再评估 cut-face AO 是否已经足够
5. 最后再决定是否做动态体积光

五、为什么要先加轨道摄像机

1. 这是最低成本验证。
2. 它能快速回答一个关键问题：
   - 现在看到的异常，到底是运行时渲染真的错了，还是只是 Scene 视图观察方式与 shader 相机不一致。
3. 如果不先排除这个变量，后面对 cut-face、shadow、volume 的判断都会有噪声。

六、通过标准

加入轨道摄像机后，验证结论按两类处理：

A. 如果 Game 视图下结果基本正常

- 结论：
  - 当前大问题首先不是运行时渲染错误
  - 先前截图中的异常有较大概率来自 Scene 视角与 Game 相机不一致
- 后续动作：
  - 继续做轻量调试
  - 然后进入软阴影实现

B. 如果 Game 视图下仍明显错误

- 结论：
  - cut-face 判定 / normal override / 阴影耦合里至少有一项存在真实逻辑问题
- 后续动作：
  - 先修切面语义
  - 再做软阴影

七、下一轮我建议的实际执行内容

经你审批后，下一轮我建议只做以下内容：

1. 添加轨道摄像机验证脚本
2. 提供最小可调参数
3. 不引入体积光
4. 不立即重写大段 shader
5. 先帮助你在 Game 视图里稳定复现并判断问题归因
